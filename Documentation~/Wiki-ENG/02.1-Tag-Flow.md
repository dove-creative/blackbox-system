# 1. Tag Flow Overview

The Blackbox tag flow attaches related objects to a single-line log or a scope-open log, so they can later be read together in the output.

If interaction recording leaves a call flow between two objects on both sides, tags mark related targets on the source log and leave auxiliary tag logs on the target object side. Therefore, a tag does not create a call range. It connects related targets to an already-recorded log line.

The center of this flow is `TagHandle`. `Write(...)` returns a `TagHandle` right after recording, and `Scope(...)` stores a `TagHandle` inside `ScopeHandle` so tags can be attached to the scope-open log.

---

# 2. Core Components

- `TagHandle`: A value that can receive `.With(...)` right after recording. If it has a valid `LogContext`, it fixes the target list on the source log. If not, it only returns the fallback message.
- `TargetTypes`: A policy that selects how much source information appears in the target-side tag log.
- `LogContext.ResolveWith(...)`: Fills the source log's `Tags` and adds tag logs to the required target objects.
- `Blackbox.Tag(...)`: Stores an auxiliary tag log on the target object side.
- `LogFormatter`: Replaces placeholders such as `%0` and `%1` with actual target display strings, and appends targets not used in the message as extra tag information.

Because of this composition, tag recording splits the responsibility between recording time and output time. At recording time, the source log and target list are stored structurally. At output time, `LogFormatter` creates the string shown to the user.

---

# 3. `.With(...)` Execution Order

1. `Write(...)` or `Scope(...)` stores the source log.
2. A `TagHandle` is created with the stored log position and fallback message.
3. The caller passes related objects to `.With(...)`.
4. If the last argument is `TargetTypes`, it is used as the target display policy and excluded from the target object list.
5. Non-null objects are converted to `Blackbox` through `BlackboxRegistry.GetBlackbox(...)`.
6. Null targets remain null, preserving the fact that the caller intentionally displayed an empty target.
7. `LogContext.ResolveWith(...)` fixes the source log's `Tags` and `TagTargetTypes`.
8. If the target is not null and differs from the source object, the target object's `Blackbox.Tag(...)` leaves an auxiliary tag log.
9. The return string from `.With(...)` is converted into the final message form through `LogContext.RenderMessage(...)`.

The important point in this order is that the source log must exist first. A tag does not start a new record. It is a follow-up step that resolves and attaches related targets to the source log that was just created.

# 4. Targets and Display Policy

`%0` and `%1` inside a message correspond to the order of targets passed to `.With(...)`. For example, the first target goes into `%0`, and the second target goes into `%1`. Targets not used in the message are appended to the end of the output as extra tag information.

A call with no targets and a null target are different.

- `.With()`: A call with no targets to connect.
- `.With(null)`: A call that explicitly connects one null target.

The display range of the target-side auxiliary tag log is controlled by `TargetTypes`.

- `TargetTypes.None`: Leaves only a minimal tag mark.
- `TargetTypes.Name`: Displays the source object name and id.
- `TargetTypes.InteractionId`: Displays the tag connection id.
- `TargetTypes.Message`: Displays the source message.
- `TargetTypes.Full`: Displays all available source information.
- `TargetTypes.BasedOnSettings`: Follows the default policy from global settings.

This policy does not control how the source log message is replaced. It controls the amount of information in the auxiliary tag log added to the target object.

---

# 5. Implementation Considerations

A tag target is not a copy of the object's state. At recording time, the target object is resolved to a `Blackbox`; at output time, the display name and id of that `Blackbox` are used.

The target-side auxiliary tag log is not an edge that expands the interaction output graph. `.With(...)` is an auxiliary connection for reading related targets together, and its role differs from a flow such as `Exert(...)`, which traces call relationships.

`TagHandle.FromMessage(...)` is a fallback path that keeps the existing message flow intact even when there is no valid recording context. In this case, calling `.With(...)` does not resolve tags and only returns the stored message.

If the global target display policy must be changed, it is safer to configure it before recording starts. After recording, policies of already-written tag logs and new logs can be mixed.
