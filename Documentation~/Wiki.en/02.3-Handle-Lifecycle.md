# 1. Handle Lifecycle Overview

Blackbox uses value-type handles to align scope closing and interaction merging with the `using` flow. `ScopeHandle` closes an opened scope, and `ExertHandle` tries to merge the receiving-side scope.

Both handles are `public readonly struct` values. Since value types can be copied, the real post-processing must happen only once even if `Dispose()` is called repeatedly on copies. `HandleManager<T>` is responsible for preventing this duplication.

The handle lifecycle can therefore be viewed in three steps.

1. The recording API receives a handle id and creates a handle.
2. The caller marks the post-processing point with `using` or direct `Dispose()`.
3. `HandleManager<T>` checks whether the id is alive, and lets only the first dispose pass into real log post-processing.

---

# 2. Scope Handle

`ScopeHandle` is the handle for closing the open log created by `Scope(...)`.

At creation time it has the following information.

- `_id`: The duplicate-prevention id issued by `HandleManager<ScopeHandle>`
- `_createdThreadId`: The thread id where the handle was created
- `_context`: The `LogContext` to receive the close request
- `_scopeId`: The scope id that connects open/close logs
- `_tagHandle`: The auxiliary value for attaching related targets to the scope-open log

When `Dispose()` is called, it first checks whether the id is valid. If the id has already been disposed or is the default value, nothing happens. If this is the first valid dispose, it checks whether output is complete and whether a context exists, warns if the creation thread and dispose thread differ, and calls `LogContext.CloseScope(...)`.

`With(...)` attaches tag targets to the scope-open log. The return value is the same `ScopeHandle`, so the caller can keep the same scope lifecycle with `using` after the tag connection.

# 3. Interaction Handle

`ExertHandle` is the handle for merging an interaction log created by `Exert(...)` with a receiving-side scope.

At creation time it has the following information.

- `_id`: The duplicate-prevention id issued by `HandleManager<ExertHandle>`
- `_createdThreadId`: The thread id where the handle was created
- `_targetContext`: The `LogContext` where the receiving-side interaction log was stored
- `_interactionId`: The id connecting both interaction logs

When `Dispose()` is called, it prevents id duplication first in the same way as `ScopeHandle`. If this is the first valid dispose, it checks whether output is complete and whether a context exists, then uses `TryMergeScope(...)` to find an open scope right after the receiving-side interaction log.

If an immediately following open scope exists, the interaction log is invalidated and the scope-open log carries the interaction information. If no matching scope exists, the interaction log remains as a one-line record.

The merge is attempted only inside the receiving-side `LogContext` fixed at creation time. It does not arbitrarily merge scopes that appear later in another thread context.

# 4. Handle id Management

`HandleManager<T>` issues ids for value-type handles and centrally manages whether they are alive. Even if the same handle value is copied, the id is the same, so only the first dispose succeeds and later disposes fail.

An id is made from a slot and a generation. The slot is a position in the management array, and the generation distinguishes an old handle from a new owner when a previously freed slot is reused.

The default implementation uses the lock-free path. If the `BLACKBOX_LOCK_HANDLES` symbol exists, it switches to an explicit lock-based implementation. Both implementations have the same external responsibility.

- `GetHandleId()`: Issues a new handle id.
- `TryDisposeHandleId(...)`: Changes a living id to the disposed state.
- `IsAlive(...)`: Checks whether an id is still alive.
- `WarnIfThreadMismatch(...)`: Warns when the creation thread and dispose thread differ.
- `Reset()`: Clears id management state at a safe initialization point.

---

# 5. Implementation Considerations

The default flow is to dispose handles on the thread where they were created. Disposing from another thread can make the recording structure differ from the intention, so Blackbox leaves a warning.

After output finishes, `Infrastructure.IsPrinted` becomes true. If remaining handles are disposed in this state, close logs or merge processing are not added.

`BlackboxHandle.ForceReset()` resets not only registry and runtime state, but also `HandleManager<T>` state. Therefore it must not be called during an execution where undisposed handles remain.

Old handles after reset are no longer reliable. If reset is needed for repeated runs or tests, call it only at a safe point where the previous recording flow has fully ended.
