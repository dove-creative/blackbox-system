## 1. Export Pipeline Overview

The Blackbox export pipeline converts recorded `LogData` into files that are easy to read. Rather than giving new meaning to the source logs, this process applies graph collection and display conversion required for output.

`BlackboxHandle.Export(...)` obtains the `Blackbox` of the root object, captures the currently issued sequence as a boundary, and calls the exporter that matches the configured output format. Logs after that boundary are not mixed into this output.

Output is divided into three major steps.

1. Collect the root object and related objects to build an export graph.
2. Convert each object's logs into an output list.
3. Write a file in text or HTML format.

## 2. Export Graph Collection

`BuildExportGraph(...)` creates an `ExportNode` list from the `Blackbox` instances to include in the output. The first node is the root object requested by the user, and later nodes are related objects found through interaction logs.

Collection follows these rules.

- A visited `Blackbox` is not visited again.
- If `recursionDepth` is 0 or greater, traversal follows only up to that depth.
- Normal focused output follows the `ExertedBy` direction and includes objects that affected the root object.
- Full output also follows the `ExertingTo` direction in addition to `ExertedBy`, including objects affected by the root object.
- Tag targets created by `.With(...)` are related display information only, so they are not used as expansion paths for the export graph.

Each `ExportNode` has the current object, parent object, output depth, and converted log list. The HTML exporter uses this information to create object-level boxes and display inclusion relationships between objects.

## 3. Log Conversion for Output

Each object included in the export graph gathers the logs from its `LogContext` instances and merges them into one output list.

The conversion flow proceeds in this order.

1. `GetLogsByContext(maxSequence)` retrieves per-thread logs up to the boundary sequence.
2. `FlattenSteps(...)` turns empty open/close pairs into logs that look like `ScopeType.Step`.
3. `ResolveScopeDepths(...)` reads the open/close flow and calculates display depth.
4. `Blackbox.MergeLogContext(...)` reorders logs from multiple threads by global sequence.

This conversion is for output only. It does not change the recording meaning of the original `LogData`; it calculates depth and one-line display form that are easier for users to read.

## 4. Text/HTML Output

`TxtExporter` passes each log to `LogFormatter.RenderTextLine(...)` and creates one-line strings. The structure is simple, but scope depth, method labels, tag replacement, and interaction arrows follow the shared rules of `LogFormatter`.

`HtmlExporter` lays out the same log data as an HTML document. Each log row contains sequence, thread information, time, method label, message, and interaction links. If the interaction peer is included in the current output, it becomes a clickable link; otherwise it remains as a disabled display.

In HTML output, tag connection ids created by `.With(...)` are not attached directly to target names inside the message body. They are displayed as orange tag badges in the form `[id]`. Therefore, the body keeps object names readable, while the badge owns the connection id and navigation behavior.

`HtmlExporter.Templates` provides the styles and behaviors required for HTML output. Screen behaviors such as folding/unfolding, moving to interaction rows, highlight effects, and reduced motion settings are gathered in this file.

`LogFormatter` is responsible for assembling the meaning of the message itself. The HTML exporter handles HTML encoding and link display, but it does not reinterpret tag replacement or method label rules.

## 5. Implementation Considerations

`LogDirectory` is the location where output files are written. The exporter creates the directory when needed, writes the file, and logs the path.

Output is a snapshot with a boundary sequence. Logs written after output starts are not mixed into the same file.

`CrashExport(...)` uses the normal output flow while adding exception information to the file name and the beginning of the body. Therefore, crash output shares the same graph collection and log conversion rules.

If `openLog` is enabled, the exporter tries to open the file after output. Even if this fails, the output file itself remains, and only failure information is logged as a warning.

After one output is finished, additional recording and duplicate output are restricted. If a new recording set is needed for repeated runs or tests, call `BlackboxHandle.ForceReset()` at a safe point.
