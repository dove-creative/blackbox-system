# 개요

객체별 활동, 실행 맥락, 객체 간 상호작용 흐름을 기록하고 추적하는 프레임워크

---

## 목차

1. Blackbox 소개
2. 기본 기록 구조
3. 상호작용 기록 방법
4. 로그의 연결 구조와 출력

---

## 1. Blackbox 소개

Blackbox는 객체의 활동 로그, 실행 범위, 객체 간 상호작용을 하나의 기록 체계 안에서 추적하는 프레임워크이다.

| 분류  | 내용                                                      |
| --- | ------------------------------------------------------- |
| 개요  | 객체별 실행 맥락과 객체 간 상호작용 흐름을 기록하는 추적 프레임워크                 |
| 목적  | 디버깅 시 기록된 상태 변화, 처리 과정, 객체 간 연결 관계를 같은 문맥에서 파악          |
| 구현  | 대상 객체에 대응되는 Blackbox를 조회하고, BlackboxHandle을 통해 기록 기능 호출 |

### 1-1. 해결하고자 하는 문제

일반적인 로그 혹은 중단점 방식 트레이싱은 원하는 지점의 상태를 확인하는 데 탁월함을 보이며, 특정 시점에 일어나는 일을 정확히 파악하는 데 효과적이다. 반면 한 객체가 어떤 처리 과정을 거쳐 현재 상태에 도달했는지, 그 과정에서 다른 객체와 어떤 연결을 만들었는지까지 이어 읽는 데에는 한계를 가진다. Blackbox는 객체별 기록 저장소에 내부 활동, 스코프 흐름, 다른 객체와의 상호작용을 저장하여 '이 객체가 어떤 과정을 거쳐 현재 상태가 되었는가'와 '그 과정에서 어떤 객체와 연결되었는가'를 함께 추적할 수 있도록 한다.

트레이싱 활용 추적과 블랙박스 기반 추적은 아래와 같은 특징을 가진다.

|     | 로그/트레이싱                 | 블랙박스                     |
| --- | ----------------------- | ------------------------ |
| 방법  | 중단점으로 실행 흐름을 실시간으로 확인   | 객체별 기록 저장소에 활동과 상호작용을 저장 |
| 강점  | 원하는 지점의 상태를 자세히 확인 가능   | 기록된 실행 맥락과 상호작용 흐름 재구성 용이 |
| 약점  | 이전 상태 변화와 실행 맥락 추적      | 사전에 기록 호출 삽입 필요          |
| 사용처 | 특정 시점에 일어나는 일을 정확히 파악하기 | 상호작용이 많거나 원인 파악이 어려운 경우  |
| 예시  | 줄 단위의 연산·대입·호출 흐름 확인    | 객체 단위의 활동·스코프·상호작용 흐름 확인 |

### 1-2. 구현 원칙

Blackbox는 기존 코드의 제어 흐름에 최대한 적게 간섭하고, 필요 시 쉽게 비활성화할 수 있는 것을 원칙으로 한다. 기록 대상 객체가 별도의 계약이나 필드를 갖지 않아도 BlackboxHandle이 대상 객체에 대응되는 Blackbox를 찾아 기록 호출을 전달하며, 런타임 설정이 꺼진 상태에서는 주요 기록 경로가 실제 로그 저장으로 이어지지 않도록 한다.

이 원칙은 다음과 같은 특징으로 나타난다.

1. 인터페이스를 강제하지 않는다.
   기록 대상 객체가 특정 인터페이스를 구현하거나 특정 클래스를 상속할 필요가 없어야 한다.
2. 도메인 객체에 기록용 필드를 강제하지 않는다.
   객체 내부에 별도 logger 필드나 Blackbox 필드를 추가하지 않아도 기록을 시작할 수 있어야 한다.
3. 런타임 스위치로 주요 기록 기능을 비활성화할 수 있다.
   `UseBlackbox`가 꺼져 있으면 주요 기록 호출은 default 또는 fallback 값으로 이어져 실제 로그 저장을 만들지 않아야 한다.

## 2. 기본 기록 구조

Blackbox의 가장 기본적인 기능은 한 객체의 실행 맥락을 그 객체 전용 로그 저장소에 쌓는 것이다.

이 장에서는 기본 로그 저장을 먼저 설명하고, 그 위에 스코프 저장, 상호작용 기록, 연결된 출력 구조가 차례로 추가되는 순서로 Blackbox를 소개한다.

### 2-1. 로그 저장 기능

로그 저장 기능은 각 객체의 활동 기록을 독립적으로 보유하는 기능이다. 이를 통해 특정 객체가 어떤 상태 변화와 처리 과정을 거쳤는지를 발생 순서대로 추적할 수 있다.

Blackbox는 '객체가 무엇을 했는가'를 로그 객체로 저장한다.

### 2-2. 스코프 저장 기능

스코프 저장 기능은 앞서 존재하는 로그 저장소 위에 추가되는 기능이다.

기본 로그 저장이 객체의 활동을 남기는 역할을 한다면, 스코프 저장은 실행 범위의 시작과 끝을 함께 기록한다. 이를 통해 하나의 처리 단계가 언제 시작되고 끝났는지를 블록 단위로 읽을 수 있다.

통상적으로 하나의 스코프는 하나의 메서드나 의미 있는 처리 단계를 기준으로 한다.

## 3. 상호작용 기록 방법

Blackbox는 한 객체의 로그를 남기는 것에 더해, 객체 사이의 상호작용 관계를 쌍방향으로 기록(Exert)하는 기능도 가진다.

이 구조는 한 객체가 다른 객체와의 연결점을 남기거나 작업을 넘기고, 연결에 참여하는 두 객체의 Blackbox가 상호작용을 양쪽 객체의 로그에 함께 남기는 방식으로 작동한다. 이때 두 로그는 같은 상호작용 번호를 공유하며, 각 객체의 개별 로그를 읽을 때 같은 상호작용 지점을 기준으로 로그 흐름을 이어갈 수 있다.

이 방식은 보내는 쪽과 받는 쪽에 같은 흐름의 흔적을 남기므로, 여러 객체가 얽힌 실행에서도 '누가 누구에게 실행을 넘겼는가'를 따라갈 수 있도록 한다.

---

## 4. 로그의 연결 구조와 출력

Blackbox의 로그는 객체별 기록과 객체 간 상호작용 기록이 연결된 구조를 가진다.

### 4-1. 로그 객체의 연결 구조

로그 구조는 아래와 같이 읽힌다.

1. 각 Blackbox는 자신의 로그 저장소를 가진다.
2. 객체 간 상호작용이 발생하면 관련 로그가 상호작용 번호로 연결된다.
3. 스코프 기록이 있으면 각 객체 내부의 실행 범위도 함께 읽을 수 있다.

이 구조가 있기 때문에 Blackbox는 단순한 로그 출력기가 아니라, 여러 객체의 실행 흐름을 연결된 상태로 다시 읽기 위한 추적 장치가 된다.

### 4-2. 로그 출력

로그 출력은 하나의 타겟 객체만 따로 떼어내는 방식이 아니라, 그 객체를 중심으로 Exert 상호작용으로 연결된 객체들의 관계를 유지한 채 함께 읽을 수 있도록 구성된다.

출력 결과에서는 먼저 타겟 객체의 로그가 중심이 되고, 그 과정에서 Exert 상호작용으로 연결된 객체들의 대응 로그가 같은 관계망 안에서 이어진다. 이 구조 덕분에 사용자는 특정 객체 하나를 기준으로 실행 흐름을 따라가면서도, 그 객체와 연결된 다른 객체들이 어떤 방식으로 관여했는지를 함께 확인할 수 있다.

이러한 방식의 로그 출력은 다음과 같은 상황에서 유용하다.

- 복잡한 상호작용이 얽힌 실행 흐름을 한 번에 보관하고 검토할 때
- 재현이 어려운 오류 상황에서 crash 직전의 로그를 함께 남길 때
- 특정 객체를 중심으로 Exert 상호작용으로 이어진 객체들의 로그를 묶어서 확인할 때
